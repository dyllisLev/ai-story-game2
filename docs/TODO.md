## 1. 요약 메모리

대화가 길어지면 토큰 비용이 증가하고 컨텍스트 한계에 도달함. 누적된 이야기를 요약하여 효율적으로 전달.

**구현 방향:**
- Gemini API로 현재까지의 대화를 요약 요청 (별도 API 호출)
- 요약 결과를 세션 문서에 `summary` 필드로 저장
- 시스템 프롬프트에 `[이전 이야기 요약]` 섹션으로 주입
- 자동 요약 트리거: N턴마다 또는 수동 버튼

**아키텍처 호환:** 가능 — 클라이언트 JS에서 Gemini API 호출, Firestore에 저장

## 2. 최근 이야기만 전달 (슬라이딩 윈도우)

전체 대화를 보내지 않고, 최근 10~20개 메시지만 전달. 이전 내용은 요약 메모리로 대체.

**구현 방향:**
- `conversationHistory` 전체를 유지하되, API 전송 시 최근 N개만 슬라이스
- 요약 메모리(#1)와 연동: `[요약] + [최근 20개 메시지]`를 전송
- 설정에서 윈도우 크기 조절 가능 (10/20/전체)

**아키텍처 호환:** 가능 — 순수 JS 로직, 기존 구조 변경 없음

**의존성:** #1 요약 메모리가 먼저 구현되어야 효과적

## 3. 스토리 암호 기능

스토리(프리셋)에 암호를 설정하여, 작성자만 수정 가능. ID를 알아도 암호 없이 수정 불가.

**구현 방향:**
- Web Crypto API로 클라이언트에서 SHA-256 해싱
- Firestore에 `passwordHash` 필드 저장
- Firestore Security Rules에서 update 시 해시 일치 검증:
  ```
  allow update: if request.resource.data.passwordHash == resource.data.passwordHash;
  ```
- 수정 시 UI에서 암호 입력 → 해싱 → 기존 해시와 비교
- 읽기(플레이)는 암호 없이 가능, 수정만 제한

**아키텍처 호환:** 가능 — Web Crypto API (브라우저 내장) + Firestore Rules
- 보안 수준: 소규모 사용에 충분. 브루트포스 위험은 낮음 (UUID 기반 ID라 스토리 자체를 찾기 어려움)
- 필요 시 Cloudflare Workers로 서버사이드 검증 업그레이드 가능

## 4. 스토리 ID / 세션 ID 분리

스토리(프리셋)와 플레이 세션을 별도 엔티티로 분리.

**데이터 모델:**
```
stories/{storyId}              # 스토리 = 세계관, 캐릭터 등 설정 묶음
├── title, worldSetting, story, characters, ...
├── passwordHash               # 수정 암호 (#3)
├── createdAt, updatedAt
└── isPublic                   # 공개 여부

sessions/{sessionId}           # 세션 = 개인의 플레이 기록
├── storyId (reference)        # 어떤 스토리 기반인지
├── messages, summary          # 대화 내역 + 요약
├── playerPreset               # 플레이어가 커스텀한 설정 (캐릭터 등)
└── updatedAt, lastPlayedAt
```

**UX 흐름:**
- 스토리 작성자: 스토리 ID 공유 → 다른 사람이 해당 스토리로 새 세션 시작
- 플레이어: 세션 ID로 자기 플레이 이력 다른 기기에서 이어하기
- 같은 스토리를 여러 사람이 각자 독립적으로 플레이

**아키텍처 호환:** 가능 — Firestore에 `stories` 컬렉션 추가, Security Rules 확장

## 5. 화면 UI 개선

스토리 작성과 플레이에 더 집중할 수 있는 UI.

**개선 방향:**
- 스토리 작성 페이지: 에디터 경험 개선 (마크다운 프리뷰, 탭 구분)
- 플레이 페이지: 소설 읽기에 최적화된 레이아웃, 불필요한 UI 최소화
- 다크/라이트 테마 전환
- 폰트/줄간격 세밀 조정
- 키보드 단축키

**아키텍처 호환:** 가능 — 순수 CSS/JS 프론트엔드 작업

---

## 6. 기본 스토리셋 자동 불러오기

운영자가 미리 설정한 기본 프리셋을 앱 시작 시 자동으로 불러오기.

**구현 방향:**
- 방법 A: Firestore `presets` 컬렉션에서 `isDefault: true`인 문서 쿼리
- 방법 B: `public/presets/default.json` 정적 파일로 제공 (더 심플)
- 비어있는 필드에만 기본값 적용, 사용자가 수정 가능
- 여러 기본셋 중 선택할 수 있는 드롭다운

**아키텍처 호환:** 가능 — 정적 파일 또는 Firestore 쿼리

---

## 7. 모바일 지원

모바일 브라우저에서 쾌적한 사용 경험.

**구현 방향:**
- 반응형 CSS (미디어 쿼리): 768px 이하에서 세로 스택 레이아웃
- 터치 최적화: 버튼 크기, 터치 타겟 48px 이상
- 모바일 키보드 대응: 입력 시 레이아웃 안정
- 사이드 패널 → 전체 화면 슬라이드 메뉴
- `visibilitychange` 저장 트리거가 모바일에서 이미 작동 (현재 설계에 포함)

**아키텍처 호환:** 가능 — 순수 CSS/JS, 추가 라이브러리 불필요
